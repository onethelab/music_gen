# WhisperX 자막 타이밍 해결 과정

## 최초 문제
stable-ts 단독 사용 시 인스트루먼트 진입 구간에서 가사 타이밍 실패 (약 23%)

---

## Phase 1: 기본 접근 시도

### 시도 1: WhisperX 순수 transcribe → align
- 결과: 10개 세그먼트 (여러 줄이 하나로 뭉침)
- 원인: VAD가 연속 보컬을 하나의 구간으로 감지
- 평가: 세그먼트 단위 타이밍은 정확하지만 줄 단위 자막 불가

### 시도 2: WhisperX에 가사 텍스트 직접 주입 (align만 수행)
- 결과: 전부 0.0초 (타이밍 없음)
- 원인: align은 세그먼트에 대략적 start/end가 있어야 범위 내에서 정밀 정렬 수행

### 시도 3: transcribe → 텍스트를 가사로 교체 → align
- 결과: 26줄 전부 타이밍 획득 (실패 0%)
- 문제: 반복 가사가 1번째 출현 위치로 끌려감
- 원인: phoneme 매칭이 동일 텍스트를 구분하지 못함

### 시도 4: Hybrid (stable-ts + WhisperX 실패 구간 보완)
- 결과: 실패율 0%
- 문제: 시간 역전/겹침, duration 비정상 (17초짜리 등)
- 근본 원인: 실제 보컬 타이밍과 일치하는지 보장 못함

---

## Phase 2: 핵심 발견

### 오해
WhisperX의 세그먼트가 뭉쳐있는 것을 "인식 실패"로 오해했음

### 사실
- 세그먼트 내부에 **word-level 타이밍이 전부 있었음** (170개 단어, 모두 start/end 보유)
- 세그먼트가 뭉친 것은 VAD의 구간 감지 단위일 뿐
- **세그먼트는 껍데기, word-level 플랫 리스트가 핵심 데이터**

---

## Phase 3: word-level 활용 시도들

### 시도 5: 세그먼트 내 단어 수 기반 배분
- 방법: 세그먼트 단어를 가사 줄 단어 수만큼 순차 슬라이싱
- 결과: v1 100점, v2 코러스 구간에서 밀림
- 원인: WhisperX 오인식 단어(boom, beep)가 단어 수를 채워서 다음 줄을 당겨옴
- 배운 것: 단어 수만으로는 코러스/오인식 구분 불가

### 시도 6: 유사도 검증 추가 (배분 후 검증)
- 방법: 단어 수 배분 후, 배분된 단어와 가사 단어의 유사도 확인
- 문제: 단어 수 배분에서 이미 줄이 할당된 후 검증 → 순서가 잘못됨
- 배운 것: **검증은 배분 전에** 해야 함

### 시도 7: 유사도 우선 배분
- 방법: 유사도를 먼저 보고 통과한 줄만 배분
- 문제: 세그먼트 내 단어 경계 ≠ 가사 줄 경계라서 유사도가 0%로 나옴
- 배운 것: 세그먼트 단위 처리는 근본적 한계가 있음

### 시도 8: 플랫 리스트 단어 1:1 순차 매칭
- 방법: 세그먼트 무시, 플랫 리스트에서 가사 단어를 순차 탐색
- 문제: "asks"↔"cracks" (sim=0.60) 같은 오매칭 → 포인터 점프 → 연쇄 밀림
- 배운 것: 짧은 단어 간 유사도가 너무 쉽게 기준을 넘김

### 시도 9: 위치 가중치 + 연속 매칭 보너스
- 방법: 텍스트 유사도 × 위치 가중치 + 연속 보너스로 최종 점수 계산
- 결과: 중반부 코러스 밀림 해결! 줄 11~20 전부 매칭 성공
- 문제: 여전히 단어 단위 매칭이라 일부 오매칭 존재
- 배운 것: 위치 가중치는 효과적이지만 단어 단위 매칭 자체의 한계

---

## Phase 4: 최종 해결

### 시도 10: 줄 전체 텍스트 유사도 (현재 최종 버전)
- 방법: 가사 줄 전체 텍스트를 WhisperX 연속 단어 구간과 비교
- 슬라이딩 윈도우로 최적 위치 탐색 (줄 단어 수 ±2 범위)
- 줄 단위로 확정 후 해당 구간 소진, 다음 줄은 그 이후부터 탐색

```
가사 줄: "Dust falls through the colored glass"
wx[0:6] 합친 텍스트: "Dust falls through the colored glass" → sim=0.95 → 확정!
wx[0:6] 합친 텍스트: "Fraction holds a name I lost The" → sim=0.15 → 기각
```

### 이 방식의 장점
- **단어 하나의 오매칭이 전체를 망치지 않음** — 줄 전체로 비교하니까
- **boom, beep 같은 오인식이 자동 회피** — 줄 텍스트와 유사도 낮으면 해당 구간 기각
- **세그먼트 경계 무관** — 플랫 리스트만 사용
- **반복 가사 구분 가능** — 순차 소진이라 시간순 보장

### 결과 (v2 기준)
- 26줄 전부 매칭 성공 (보간 0줄)
- 타이밍 범위 2.5~10초 (대부분 3~7초)
- 이전 방식 대비 코러스 밀림 해결

---

## 기타 개선사항

### VAD onset 조정
- 기본값 0.5 → **0.3으로 변경**
- 효과: 속삭이는 보컬 구간(171초)이 감지됨
- 부작용: boom 같은 오인식 약간 증가 (줄 유사도 매칭에서 자동 필터)

### Demucs 모델 변경
- htdemucs_ft (4소스) → **htdemucs_6s (6소스)** (코드 반영 완료)
- 효과: v2 중반부 보컬 분리 개선 (WhisperX 인식률 향상)
- 비용: 처리 시간 약간 증가 (10초 → 15초, GPU 기준)

### 후처리
- 최소 duration 2.5초 강제
- 순차 강제 (다음 줄 start > 현재 줄 end)
- 매칭 실패 줄은 앞뒤 기준 보간

---

## 파이프라인 최종 구조

```
[모델 1회 로딩: Demucs htdemucs_6s + WhisperX large-v3 + align model]
  ↓
원본 mp3 (곡별 반복)
  → Demucs htdemucs_6s (GPU, 재사용) → 보컬 분리 → GPU 메모리 해제
    → WhisperX transcribe (VAD onset=0.3, 재사용) → 세그먼트
      → WhisperX align (재사용) → word-level 플랫 리스트
        → 줄 전체 텍스트 유사도 매칭 (슬라이딩 윈도우)
          → 가사 줄별 start/end 확정
            → Gemini 번역 → 이중언어 SRT
              → equalizer.py (독립 모듈) → 이퀄라이저
                → ffmpeg (preset fast, 자막 + 이퀄라이저) → mp4
```
