Suno 자동화 파이프라인에서 발생했던 오류와 해결 방법을 기록한다.

## 1. Suno fill() 입력이 안 되는 문제

### 증상
- Playwright의 `fill()` 메서드로 Style of Music, Lyrics 필드에 텍스트를 입력해도 실제로 입력되지 않음
- 코드상으로는 성공한 것처럼 보이지만 Suno 페이지에는 빈 필드로 남아있음
- Create 버튼이 disabled 상태로 클릭 불가

### 원인
- Suno는 React 기반 UI를 사용하며, React의 textarea는 내부 state로 값을 관리함
- Playwright의 `fill()`은 DOM의 value만 변경하고 React의 내부 state를 업데이트하지 않음
- React가 자체 synthetic event 시스템을 사용하기 때문에 일반적인 input 이벤트로는 state가 변하지 않음

### 해결 방법
`fill()` 대신 **클립보드 붙여넣기(ClipboardEvent)** 방식을 사용한다:

```python
def paste_into(element, text):
    element.click()
    page.wait_for_timeout(300)
    page.keyboard.press("Control+A")
    page.keyboard.press("Backspace")
    page.wait_for_timeout(200)
    # ClipboardEvent로 붙여넣기
    page.evaluate("""
        (text) => {
            const el = document.activeElement;
            const dt = new DataTransfer();
            dt.setData('text/plain', text);
            const evt = new ClipboardEvent('paste', {
                clipboardData: dt, bubbles: true, cancelable: true
            });
            el.dispatchEvent(evt);
        }
    """, text)
    page.wait_for_timeout(500)
    # 검증 후 실패 시 native setter 폴백
    actual = element.input_value()
    if len(actual) < len(text) // 2:
        page.evaluate("""
            (text) => {
                const el = document.activeElement;
                const nativeSetter = Object.getOwnPropertyDescriptor(
                    window.HTMLTextAreaElement.prototype, 'value'
                ).set;
                nativeSetter.call(el, text);
                el.dispatchEvent(new Event('input', { bubbles: true }));
                el.dispatchEvent(new Event('change', { bubbles: true }));
            }
        """, text)
```

### 향후 대응
- React 기반 웹앱에서 입력이 안 되면 `fill()` 대신 ClipboardEvent 또는 `keyboard.type()` 사용
- native setter 폴백을 항상 준비해둘 것

## 2. headless=True에서 곡 생성이 안 되는 문제

### 증상
- `headless=True`로 Playwright를 실행하면 Create 버튼을 클릭해도 곡이 생성되지 않음
- 5분 이상 대기해도 새로운 곡 링크가 나타나지 않음
- `headless=False`로 변경하면 10초 내에 정상 생성됨

### 원인
- Suno가 봇 감지(bot detection)를 구현하여 headless 브라우저를 차단함
- headless 모드에서는 `navigator.webdriver` 등의 속성으로 자동화를 감지함

### 해결 방법
```python
browser = playwright.chromium.launch_persistent_context(
    user_data_dir=chrome_profile_path,
    headless=False,  # 반드시 False
    # ...
)
```

### 향후 대응
- Suno 자동화는 항상 `headless=False`로 실행
- 다른 사이트에서도 자동화가 안 되면 headless 모드를 먼저 의심할 것

## 3. CDN URL 변경으로 mp3 다운로드 실패 (403 Forbidden)

### 증상
- `https://cdn1.suno.ai/{song_id}.mp3` URL로 다운로드 시 403 Forbidden 오류
- 이전에는 정상 작동했으나 갑자기 실패

### 원인
- Suno가 CDN URL 패턴을 변경함
- 기존: `cdn1.suno.ai/{id}.mp3`
- 변경: `audiopipe.suno.ai/?item_id={id}`

### 해결 방법
```python
download_url = f"https://audiopipe.suno.ai/?item_id={song_id}"
headers = {
    "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36"
}
response = requests.get(download_url, headers=headers)
```

### 향후 대응
- mp3 다운로드가 403/404로 실패하면 Suno 웹페이지에서 실제 다운로드 URL을 확인
- 브라우저 개발자 도구 Network 탭에서 mp3 요청 URL 패턴을 확인할 것
- User-Agent 헤더를 항상 포함

## 4. Studio 페이지 타임아웃

### 증상
- `page.goto("https://suno.com/studio", wait_until="networkidle")` 에서 타임아웃 발생
- Studio 페이지가 계속 리소스를 로드하여 networkidle 상태에 도달하지 않음

### 원인
- Suno Studio 페이지는 실시간 업데이트, 웹소켓 등으로 인해 네트워크 요청이 끊이지 않음
- `networkidle`은 500ms 동안 네트워크 요청이 없어야 하는데 이 조건이 충족되지 않음

### 해결 방법
```python
page.goto("https://suno.com/studio", wait_until="domcontentloaded")
```

### 향후 대응
- SPA(Single Page Application)에서는 `networkidle` 대신 `domcontentloaded` 사용
- 필요한 요소가 나타날 때까지 `wait_for_selector()`로 대기하는 것이 더 안정적

## 5. 곡 ID 매칭 오류

### 증상
- 생성된 곡의 제목 텍스트로 매칭하면 이전에 만든 다른 곡이 잘못 선택됨
- `title.split()[-1]` 방식의 텍스트 매칭이 불안정

### 원인
- Suno Studio에 같은 단어가 포함된 곡이 여러 개 존재할 수 있음
- 텍스트 기반 매칭은 신뢰할 수 없음

### 해결 방법
Create 버튼 클릭 **전후**의 곡 링크 ID를 비교하여 새로 생성된 곡을 식별:

```python
# Create 전: 기존 곡 ID 수집
before_ids = set()
for link in page.query_selector_all('a[href*="/song/"]'):
    href = link.get_attribute("href")
    song_id = href.split("/song/")[-1]
    before_ids.add(song_id)

# Create 후: 새로운 ID 찾기
after_ids = set()
for link in page.query_selector_all('a[href*="/song/"]'):
    href = link.get_attribute("href")
    song_id = href.split("/song/")[-1]
    after_ids.add(song_id)

new_ids = after_ids - before_ids
```

### 향후 대응
- 텍스트 매칭 대신 항상 ID 기반 비교를 사용
- 전후 비교(before/after diff) 패턴이 가장 안정적

## 6. 0바이트 mp3 다운로드

### 증상
- 다운로드된 mp3 파일이 0바이트
- 재생 불가

### 원인
- 곡 생성이 실제로 실패했거나, 아직 생성 중인 곡의 ID로 다운로드를 시도
- 잘못된 곡 ID로 다운로드 요청

### 해결 방법
- 곡 생성 성공을 확인한 후 다운로드 (위 #5의 ID 비교 방식 사용)
- 다운로드 후 파일 크기 검증:

```python
if os.path.getsize(mp3_path) < 1000:
    print("다운로드 실패: 파일이 너무 작음")
```

### 향후 대응
- 다운로드 후 반드시 파일 크기를 확인
- 1KB 미만이면 실패로 간주하고 재시도

## 7. Title 입력 필드를 찾을 수 없는 문제

### 증상
- `input[type="text"]`에서 placeholder로 검색해도 Title 필드를 못 찾거나, 찾아도 `click()` 시 타임아웃
- "Title 입력 필드를 찾을 수 없습니다" 메시지 출력

### 원인
- Suno Create 페이지에 Title input이 **2개** 존재 (Simple 모드용 + Advanced 모드용)
- Simple 모드용은 `visibility: hidden`, Advanced 모드용은 `visibility: visible`
- Playwright의 `click()`은 `visibility: hidden` 요소를 클릭하려다 타임아웃
- `visibility: visible`인 요소도 React input이라 `keyboard.type()`으로는 state가 반영 안 됨

### 해결 방법
`visibility: visible`인 Title input을 찾아 **native setter**로 값 설정:

```python
title_result = page.evaluate("""
    (title) => {
        const inputs = document.querySelectorAll('input[placeholder="Song Title (Optional)"]');
        for (const inp of inputs) {
            if (window.getComputedStyle(inp).visibility === 'visible') {
                const nativeSetter = Object.getOwnPropertyDescriptor(
                    window.HTMLInputElement.prototype, 'value'
                ).set;
                nativeSetter.call(inp, title);
                inp.dispatchEvent(new Event('input', { bubbles: true }));
                inp.dispatchEvent(new Event('change', { bubbles: true }));
                return inp.value;
            }
        }
        return '';
    }
""", title)
```

### 향후 대응
- Suno의 모든 입력 필드(textarea, input)는 React 컴포넌트이므로 **항상 native setter** 사용
- 동일 selector의 요소가 여러 개일 때 `visibility` 또는 `display`로 활성 요소 필터링

## 8. mp4 영상 길이가 실제 mp3와 불일치 (VBR mp3 duration 오류)

### 증상
- mp4 영상이 실제 음악보다 훨씬 길게 생성됨 (끝부분 무음)
- 또는 mutagen 기준으로 자르면 음악이 중간에 끊김
- Suno에서 재생한 길이와 mp4 재생시간이 다름

### 원인
- Suno가 생성하는 mp3는 **48kHz VBR** 인코딩
- moviepy의 `AudioFileClip`은 기본 `fps=44100`으로 디코딩하며, 내부 ffmpeg가 VBR mp3의 duration을 **bitrate 기반으로 추정**하여 실제보다 길게 계산
- mutagen은 mp3 프레임 기반으로 계산하나 VBR 파일에서 **실제보다 짧게** 계산하는 경우 있음
- 세 도구가 각각 다른 duration을 반환:
  - moviepy: 가장 길게 (잘못된 추정, 끝부분 무음 패딩)
  - mutagen: 가장 짧게 (프레임 계산 오차, 음악 끊김)
  - ffmpeg 실제 디코딩: 정확한 값

### 해결 방법
ffmpeg로 mp3를 **끝까지 디코딩**하여 실제 duration을 측정한 후, moviepy 추정값이 다르면 보정:

```python
import subprocess
from moviepy.config import FFMPEG_BINARY

# ffmpeg로 실제 디코딩된 duration 측정
result = subprocess.run(
    [FFMPEG_BINARY, '-i', mp3_path, '-f', 'null', '-'],
    capture_output=True, text=True
)
total_duration = None
for line in result.stderr.split('\n'):
    if 'time=' in line:
        m = re.search(r'time=(\d+):(\d+):(\d+\.\d+)', line)
        if m:
            total_duration = int(m.group(1)) * 3600 + int(m.group(2)) * 60 + float(m.group(3))

audio = AudioFileClip(mp3_path)
if total_duration and abs(audio.duration - total_duration) > 1:
    audio = audio.subclipped(0, total_duration)
```

### 향후 대응
- Suno mp3는 항상 VBR + 48kHz일 수 있으므로, moviepy의 `audio.duration`을 신뢰하지 말 것
- mutagen도 VBR에서는 부정확할 수 있으므로 신뢰하지 말 것
- **ffmpeg 실제 디코딩 (`-f null -`)** 결과만 신뢰
- `video_create.py`에 이미 보정 로직 적용 완료

## 9. mp3 다운로드가 잘리는 문제 (렌더링 미완료)

### 증상
- 다운로드된 mp3가 Suno 사이트의 원본보다 짧음 (예: 2:16 vs 2:39)
- 곡 끝부분이 잘려서 재생됨

### 원인
- `create_song()`에서 새 곡 링크 출현만 감지하고 5초 후 바로 다운로드
- Suno는 곡 링크를 먼저 만든 뒤 **백그라운드에서 렌더링을 계속** 진행함
- CDN이 아직 렌더링 중인 불완전한 파일을 그대로 내려줌

### 해결 방법
1. 곡 감지 후 대기 시간을 **60초**로 증가
2. 다운로드 시 **파일 크기 안정화 확인** (동일 크기 2회 연속 시 완료 판정)

```python
# 1. 곡 감지 후 60초 대기
safe_print("렌더링 완료 대기 중... (60초)")
page.wait_for_timeout(60000)

# 2. 파일 크기 안정화 확인 (최대 5회, 15초 간격)
prev_size = 0
for attempt in range(5):
    # 다운로드
    file_size = os.path.getsize(filepath)
    if file_size > 0 and file_size == prev_size:
        break  # 크기 안정 = 렌더링 완료
    prev_size = file_size
    time.sleep(15)
```

### 향후 대응
- Suno 곡 생성 후 최소 60초 대기 후 다운로드
- 다운로드 파일 크기가 2회 연속 동일하면 렌더링 완료로 판정

## 10. 유튜브 제목이 "V1", "V2"로만 업로드되는 문제

### 증상
- 유튜브에 업로드된 영상 제목이 곡 제목 없이 "V1", "V2"로만 표시됨

### 원인
- `youtube_upload.py`의 `parse_youtube_script()`가 `## 유튜브 제목` 헤딩을 찾음
- 유튜브 스크립트 파일에서 `## 제목` 등 다른 이름으로 작성하면 파싱 실패 → title이 빈 문자열
- `find_upload_targets()`에서 title에 version_label을 추가: `"" + " V1"` → `" V1"`

### 해결 방법
유튜브 스크립트 md 파일의 헤딩을 정확히 맞춘다:

```markdown
## 유튜브 제목     ← 반드시 이 이름
(제목 내용)

## 유튜브 설명     ← 반드시 이 이름
(설명 내용)

## 태그            ← 반드시 이 이름
(쉼표 구분 태그)
```

### 향후 대응
- `08_youtube_script/` 파일 작성 시 반드시 `## 유튜브 제목`, `## 유튜브 설명`, `## 태그` 헤딩 사용
- 참조 형식: `08_youtube_script/13_Expiration_Date.md`

## 요약: 문제 발생 시 체크리스트

1. **입력이 안 됨** → `fill()` 대신 ClipboardEvent/paste 또는 native setter 사용
2. **곡 생성 안 됨** → `headless=False` 확인
3. **다운로드 403/404** → CDN URL 패턴 변경 확인, User-Agent 헤더 추가
4. **페이지 타임아웃** → `networkidle` 대신 `domcontentloaded` 사용
5. **잘못된 곡 선택** → 텍스트 매칭 대신 ID 전후 비교
6. **0바이트 파일** → 생성 완료 확인 후 다운로드, 파일 크기 검증
7. **Title 입력 안 됨** → `visibility: visible` 필터링 + native setter 사용
8. **mp4 길이 불일치** → ffmpeg 실제 디코딩으로 duration 측정 (`-f null -`)
9. **mp3 잘림** → 곡 감지 후 60초 대기 + 파일 크기 안정화 확인
10. **유튜브 제목 누락** → `## 유튜브 제목` 헤딩 확인 (`## 제목` 등 불가)
